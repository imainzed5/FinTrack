This design was derived from the current running system behavior in:
- `src/lib/types.ts`

## What is persisted today (source of truth)

1. Transactions (including split lines, recurring templates, generated occurrences, tags, notes, receipt payload)
2. Budgets (month + category/sub-category + limit + rollover + threshold flags)
3. Derived analytics are computed on read (dashboard/insights/savings/timeline)
4. Offline sync uses client-generated UUIDs, then pushes to server

## Proposed relational model

1. `profiles`
- One row per Supabase Auth user.
- Stores user-level defaults like `currency_code` and `timezone`.

2. `user_settings`
- Theme and notification preferences.

3. `transactions`
- Main financial events table.
- Keeps primary category fields for fast list views.
- Supports recurring template metadata (`recurring_frequency`, `recurring_next_run_at`, etc.).
- Stores sync marker (`synced`) to preserve offline semantics.

4. `transaction_splits`
- Normalized split allocations per transaction.
- Constraint trigger enforces split total = transaction amount and minimum split lines.

5. `budgets`
- Month-scoped budget records.
- Supports overall and category/sub-category budgets.
- Preserves `alert_thresholds_triggered` array for compatibility with current app response shape.

6. `budget_threshold_alerts`
- Event history for thresholds (50/80/100) with payload required for notifications and auditing.

## Why this model fits your current app

1. Supports current API filters immediately:
- `/api/transactions?month=YYYY-MM`
- `/api/transactions?category=...`
- budget lookups by month and category

2. Preserves behavior from existing logic:
- recurring generation via `recurring_next_run_at`
- split validation and category allocations
- threshold trigger tracking

3. Adds production constraints missing from JSON storage:
- FK integrity
- unique budget scope per month
- value checks for money, thresholds, month normalization
- RLS for per-user data isolation in Supabase

## Data mapping from current TypeScript model

1. `Transaction`
- `id` -> `transactions.id`
- `amount` -> `transactions.amount`
- `category` -> `transactions.category`
- `subCategory` -> `transactions.sub_category`
- `merchant` -> `transactions.merchant`
- `description` -> `transactions.description`
- `date` -> `transactions.transaction_at`
- `paymentMethod` -> `transactions.payment_method`
- `notes` -> `transactions.notes`
- `tags` -> `transactions.tags`
- `attachmentBase64` -> `transactions.attachment_base64`
- `recurring.*` -> `transactions.recurring_*`
- `recurringOriginId` -> `transactions.recurring_origin_id`
- `isAutoGenerated` -> `transactions.is_auto_generated`
- `synced` -> `transactions.synced`
- `split[]` -> `transaction_splits`

2. `Budget`
- `id` -> `budgets.id`
- `category` -> `budgets.category`
- `subCategory` -> `budgets.sub_category`
- `monthlyLimit` -> `budgets.monthly_limit`
- `month` -> `budgets.budget_month` (first day of month)
- `rollover` -> `budgets.rollover`
- `alertThresholdsTriggered` -> `budgets.alert_thresholds_triggered`

## Notes on future-proofing

1. Insights/timeline stay computed-on-demand for now, matching current code.
2. If volume grows, add materialized views for monthly aggregates.
3. For recurring generation, run a scheduled Supabase Edge Function that processes due templates by `recurring_next_run_at`.
